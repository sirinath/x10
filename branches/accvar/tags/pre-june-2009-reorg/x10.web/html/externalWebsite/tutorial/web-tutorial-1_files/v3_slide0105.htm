Single-threaded vs. Multi-threaded GUI frameworks
1) Java approach -- Single-threaded GUI framework
– GUI objects are kept consistent by thread confinement
– Pro: Programmer does not have to worry about deadlock in GUI thread
– Cons:
• Cannot exploit parallelism to speed up GUI framework
• Reasoning about data accesses across task boundaries can still be tricky due
to nondeterminism of task scheduling
2) X10 approach – Single-place Multi-threaded GUI framework a
– All GUI tasks are scheduled at GuiExecutorPlace -- GUI objects are accessed
only by activities in GuiExecutorPlace
– Pro: Can easily exploit parallelism within GuiExecutorPlace
– Con: atomic blocks necessary to ensure mutual exclusion among tasks (but
making atomicity explicit should also make the code more maintainable?)

Multi-threaded GUI framework
