This slide summarizes the basic concepts in the X10 programming models. A more detailed technical description can be found in the OOPSLA 2005 paper titled “X10: an object-oriented approach to non-uniform cluster computing”.
•All concurrency is expressed as asynchronous activities.  Our experience has been that this single notion of an asynchronous activity can subsume many levels of parallelism that  a programmer may encounter e.g., threads, structured parallelism (including OpenMP), messaging (including MPI), DMA transfers, etc.  In the above slide each vertical green rectangle represents the stack for a single activity.
•
•Places encapsulate an affinity between activities and data. All data that is writeable is assumed to be partitioned across places, thereby resulting in a Partitioned Global Address Space (PGAS).  For example, we see a number of activities and local objects in Place 0 above.  In addition, Place 0 can hold a local section of a global array.  An activity may hold references to remote objects i.e., objects located at places different from the activity’s place.  If an activity attempts to operate on a remote object, then it does so by logically creating a new activity at the remote place to perform the operation as indicated by the arrows labeled “Outbound Activities” and “Inbound Activities”.  Immutable (readonly) data has a special status because it can be accessed freely from any place.
 
A place is a virtual concept.  While it is natural to think of a place mapping to a single SMP node (say), it may be possible to map multiple places on the same SMP node and even migrate the places within and across SMP’s if it’s advantageous to do so.
3) Atomic blocks are used by multiple activities in the same place to coordinate mutual exclusion in their accesses to shared mutable data.  A key benefit of atomic compared to classical acquire/release or monitor-enter/monitor-exit is that the programmer does not need to specify a lock to be used for reinforcing mutual exclusion.
There are three possible approaches for implementing atomic blocks currently being pursued in the X10 project:
a)Automatic lock assignment by the system.  The approach here is to let the static/dynamic compiler study the data being accessed in each atomic blocks, and assign locks from a compiler-managed set to different atomic blocks so as to guarantee mutual exclusion.
b)Use of transactional memory.  Any transactional memory implementation (hardware, software, or hybrid) can be used to implement atomic blocks.
c)Use of nonblocking instructions in hardware.  In many cases, the body of the atomic block is simple enough that it may be possible to implement the entire atomic block using nonblocking instructions in the target processor.
d)
4) Finish is the X10 operation that is used to detect termination of a set of subactivities created by an X10 activity (the set is determined by the scope of the Finish construct)
5) An important result is that any X10 program written with the above constructs can never deadlock.  There are additional constructs in X10 that have not been described here, for the sake of brevity.

Clocks are probably most complex synchronization construct in X10.
- Basically a split barrier: contract between a varying set of threads to "meet" at a certain point in their operations -
   communicate data - then to continue (phased computation)
-Clock support in the language for (at least) two reasons:
-  language restricts the use of this blocking synchronization construct such that deadlock cannot arise - there are "clocked" data structures that are constant during a phase, but might be updated at phase transitions - a nice way to "resolve data races" (parallel computation is determinate, because one writer - multi reader) -- I will not talk about this in more detail here. -More important in the context of typing and type systems are the restrictions on the use of clocks (static and dynamic semantics)   Last comment for dynamic semantics is to avoid race between block phase advance and registration.

Problem
--------
Perform a MonteCarlo simulation nRunsMC time. A tasks array (of size nRunsMC) is initialized with
ToTask instances at each index. The task is to simulate the trajectory of a stock, and compute
its expected rate of return and volatility. The overall simulation reports the average
of the expected rate of return and volatility.
Parallelization
-----------------
Each task instance can be run in parallel, with the values of expected rate of
return and volatility placed in an array. The sequential version executes these
tasks (and computes the required metrics) in sequence. The parallel version
distributes the array into a single place and uses a finish/ateach to compute
each task in parallel. It uses a collective operation (sum) to compute the
average. The distributed version differs only in that the array is block distributed
across the given number of places. 
Thus the change in code from sequential to parallel, and from parallel to distributed
is minimal (one line).
